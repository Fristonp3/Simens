# Project 工程编译与烧录教程（GD32F450 / CMake / OpenOCD / ST-Link）

本文适用于当前工程目录：

`/home/fristonp/Simens/Project`

开始前先注意两点：

- 工程目录名是 `Project`，但 `CMakeLists.txt` 里的目标名仍是 `gd32f450_template`，所以生成文件仍叫 `gd32f450_template.elf/.hex/.bin/.map`。
- 当前仓库里的 `scripts/*.sh` 默认没有执行权限。下面优先给出可直接运行的命令；如果你想使用 `./scripts/...` 形式，请先执行 `chmod +x scripts/*.sh`。

---

## 1. 环境准备

进入工程目录：

```bash
cd /home/fristonp/Simens/Project
```

如果你在 WSL / Ubuntu 下第一次配置环境，可安装依赖：

```bash
bash scripts/setup_wsl_env.sh
```

安装完成后建议确认版本：

```bash
cmake --version
ninja --version
arm-none-eabi-gcc --version | head -n 1
openocd --version | head -n 1
```

## 2. SDK 目录检查

本工程默认要求 GD32 SDK 根目录为：

`/home/fristonp/Simens`

该目录下至少应包含：

- `Firmware/`
- `Utilities/`

检查命令：

```bash
cd /home/fristonp/Simens/Project
bash scripts/check_gd32_sdk.sh /home/fristonp/Simens
```

若输出 `GD32 SDK check passed`，说明 SDK 路径正确。

---

## 3. 编译固件

本文示例统一使用 `build/` 作为构建目录；如果你想沿用仓库里已有的 `build_software_scheme/`，把后续命令中的 `build` 替换成 `build_software_scheme` 即可。

### 3.1 推荐方式：直接用 CMake 编译

这组命令和 `scripts/configure_build.sh` 的实际配置一致，并且已按当前 `Project` 工程验证通过：

```bash
cd /home/fristonp/Simens/Project
env -u CC -u CXX -u CFLAGS -u CXXFLAGS -u CPPFLAGS -u LDFLAGS \
  cmake --fresh -S . -B build -G Ninja \
  -DCMAKE_TOOLCHAIN_FILE=cmake/toolchain-arm-none-eabi.cmake \
  -DGD32_SDK_ROOT=/home/fristonp/Simens

cmake --build build
```

### 3.2 脚本方式

如果你希望直接使用仓库脚本，先给脚本加执行权限：

```bash
cd /home/fristonp/Simens/Project
chmod +x scripts/*.sh
./scripts/configure_build.sh /home/fristonp/Simens build
```

编译成功后会在 `build/` 目录下生成：

- `gd32f450_template.elf`
- `gd32f450_template.hex`
- `gd32f450_template.bin`
- `gd32f450_template.map`

## 4. 使用 ST-Link 烧录

完成编译后执行：

```bash
cd /home/fristonp/Simens/Project
cmake --build build --target flash
```

该命令内部等价于：

```bash
openocd -f /home/fristonp/Simens/Project/openocd_stlink.cfg \
  -c "program /home/fristonp/Simens/Project/build/gd32f450_template.elf verify reset exit"
```

补充说明：

- `flash` 目标只有在执行 CMake 配置时已经安装了 `openocd` 才会生成。
- 如果 `cmake --build build --target flash` 提示目标不存在，先安装 `openocd`，然后重新执行一次 CMake 配置或重新跑一遍第 3 节的编译命令。

## 5. WSL 下 ST-Link 常见问题

### 5.1 报错 `Error: open failed`

这通常表示 OpenOCD 无法打开 ST-Link USB 设备，优先检查 USB 透传和权限。

先看 WSL 是否识别设备：

```bash
lsusb | grep -i -E "stlink|st-link|stmicro|0483"
```

最小连接测试：

```bash
openocd -f interface/stlink.cfg -f target/stm32f4x.cfg -c "init; shutdown"
```

如果 WSL 看不到设备，在 Windows 管理员 PowerShell 中执行：

```powershell
usbipd list
usbipd bind --busid <BUSID>
usbipd attach --wsl --busid <BUSID>
```

然后回到 WSL 再测试 `lsusb` 和 OpenOCD。

如果能看到设备但仍然 `open failed`，可先用 root 验证是否为权限问题：

```bash
sudo openocd -f /home/fristonp/Simens/Project/openocd_stlink.cfg \
  -c "program /home/fristonp/Simens/Project/build/gd32f450_template.elf verify reset exit"
```

如果连接不稳定，可以把 `openocd_stlink.cfg` 中的 `adapter speed` 从 `4000` 调低到 `1000` 再试。

## 7. 说明

- 链接阶段出现 `_write`、`_read`、`_close`、`_isatty` 之类 `newlib` 警告通常不影响生成 `.elf/.hex/.bin`，当前工程实测可正常编译完成。
- 工程里虽然有 `bootloader/` 目录，但当前 CMake 只生成一个 `gd32f450_template.elf` 镜像，没有单独的 bootloader 构建产物。
- 如果你把构建目录改成了别的名字，烧录命令里的 `.elf` 路径也要同步改成对应目录。
